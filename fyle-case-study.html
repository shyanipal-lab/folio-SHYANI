The Setup

What even is Fyle?

Fyle is a B2B expense management app used by companies whose employees travel regularly for work. Think client visits, daily commutes, site visits. The app lets employees log expenses and get reimbursed without drowning in paperwork.

The mileage feature let users add a route and get reimbursed for the distance traveled. Sounds simple. It wasn't. Every single mileage entry required the user to go through the full form — date, vehicle, cost centre, project, route — one trip at a time.

For someone commuting every day? That's filling out the same form 20+ times a month. No wonder the support inbox was on fire.

The Problem

The old flow was a patience-testing endurance sport

Users had to add each mileage expense individually. Which sounds fine until you realise these are people who commute every single working day.

😨
One expense at a time
Every single trip required opening the form, filling all fields, submitting, then starting again for the next one. For a daily commuter, this meant repeating the process ~20 times a month.
🧐
Repetitive data entry
Each entry needed: Date, Vehicle Type, Cost Center, Project, and Route. The same fields. Every. Single. Time. Users were basically punished for traveling consistently.
🥱
No memory of what's been done
Users had to mentally track how many expenses they'd added and how many were left. There was no overview, no count — just a screen of input fields with no sense of progress.
🫠
No visibility into status
To see which expenses were saved vs reported, users had to navigate all the way back to the reports page. No glanceable summary. No peace of mind.
"The user had to add one expense at a time, remember how many they'd added, and had no overview — just a screen of multiple input fields."
— Shyani's problem summary, based on 500+ support tickets

Auto-fill the date. Since 93% travel daily, the date is almost always today. Don't make them type it.

Show all entries at once. Users need to see the full picture — how many expenses, which are saved, which are reported — without bouncing between screens.

Reduce repeated fields. Vehicle type, cost centre, and project are often the same across trips. Treat repeated data as a constant, not a question.
The Transformation

Before the feature vs after

Before
  • One mileage expense added per session
  • All fields manually entered every time
  • Date had to be typed in each entry
  • No way to see all expenses at a glance
  • Had to navigate to reports page for status
  • 500+ support tickets a month. Someone was losing their mind.
After
  • Multiple mileage expenses added in one session
  • Constant fields (vehicle, cost centre) set once
  • Date auto-filled — editable if needed
  • Table overview of all expenses and their status
  • Save or report all expenses in a single action
  • Complaint tickets dropped dramatically. Peace restored.

Key Design Decisions

Four calls that made the feature work

01
Auto-filled date with amendment option
Since 93% of users travel daily, the date defaults to today. But the amendment option is front-and-centre — not buried — for the 7% who batch-enter expenses later.
02
Constant input fields
Vehicle type, cost centre, and project are set once at the top and applied to all rows. Stops users from re-entering the same data 20 times. This alone probably saved hours per user per month.
03
Table view over form stacking
Instead of a form that grows infinitely downward, all entries live in a scannable table. Users can see everything at once: what's saved, what's pending, what needs attention.
04
Bulk save or report action
One action at the end covers all entries. No more submitting one by one. The success state confirms everything went through cleanly — no ambiguity, no second-guessing.

Usability Testing

We broke it so users didn't have to

The prototype was tested via Maze with teammates and peers. After the first round of testing, two issues surfaced:

Issue 1: Users were confused about whether the date field referred to when the expense was incurred or when it was being reported. Fixed with clearer labelling.

Issue 2: Adding multiple cost centres, projects, and vehicle types simultaneously caused errors. The UI wasn't clear enough about scope. Fixed by making the "constant" logic more visually obvious.

Iterated, retested, shipped. Classic.

Competitive Analysis

How Fyle stacked up

We analysed direct and indirect competitors — Zoho Expense and Expensify — to understand the market baseline.

✅
Number input field — Fyle's new feature. Competitors didn't have a dedicated bulk entry mode.
✅
Auto-filled date — Unique to this design. Neither Zoho nor Expensify defaulted to today's date intelligently.
✅
Auto-check for reimbursement — Reduced a decision users had to make every time.
✅
Inline expense removal — Icon-level delete right in the table row, no navigation required.
Takeaways

What this project taught me

01
Listen to the tickets, not just the users
500 support tickets are 500 user interviews you didn't have to schedule. The signal was right there. The solution was in understanding the pattern, not just the complaints.
02
One insight can unlock the whole design
The 93% daily-travel stat reframed everything. Auto-filling the date wasn't a small UX tweak — it was the central insight that made bulk entry feel effortless instead of just "less bad".
03
Consider all stakeholders — including support
This project involved product, engineering, and the support team. The customer support team had the richest data on user pain. Looping them in earlier would have accelerated the research phase.
04
Confusion in testing is a gift
The date field confusion found in Maze testing was a real problem — and catching it before shipping saved users from making errors in their actual expense reports. Test early, break it yourself first.
